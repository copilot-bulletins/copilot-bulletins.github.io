Biweekly #2 · March 23 to April 6, 2026 · Admins Edition

🎯 This edition's thread

The past two weeks mark a clear turning point for Copilot governance at scale. GitHub has moved several organization-level controls from preview to general availability, and introduced new mechanisms for managing the cloud agent's security posture, runner infrastructure, and usage visibility across the entire organization — all configurable from a single place.

⚠️ Heads up

  • GPT-5.1 Codex, GPT-5.1-Codex-Max, and GPT-5.1-Codex-Mini deprecated — Deadline: April 1, 2026 (already in effect) — Enable access to GPT-5.3-Codex as a replacement through your org model policies in Copilot settings. No action is needed to remove the deprecated models.

🚀 Take action

  • Enable GPT-5.3-Codex for your users by reviewing Copilot model policies in your organization settings.
  • Set organization-wide Copilot behavior by configuring organization custom instructions in your org's Copilot settings.
  • Review and lock the cloud agent's runner type and firewall settings from your organization's Copilot → Cloud agent settings.
  • Monitor individual CLI adoption using the per-user Copilot usage metrics API.
  • Questions or need support? Reach out to your GitHub team, we're here to help.

🏆 Top 5 updates

1. Organization custom instructions are now generally available

Org admins can now set default instructions that guide Copilot's behavior across all repos — applied consistently to Copilot Chat on GitHub.com, Copilot code review, and Copilot cloud agent. No more repo-by-repo configuration for common guidelines.

→ Changelog: Copilot organization custom instructions are generally available

2. Org-level firewall controls for Copilot cloud agent

Admins can now enable or disable the agent firewall across all repositories, manage the recommended allowlist, add org-wide custom allowlist entries (e.g., an internal package registry), and control whether repos can add their own entries — replacing per-repo configuration.

→ Changelog: Organization firewall settings for Copilot cloud agent

3. Org-level runner controls for Copilot cloud agent

Organization owners can now set a default runner type for all cloud agent sessions and optionally lock the setting so individual repos can't override it. Enables consistent enforcement of self-hosted runners or larger hosted runners across the org.

→ Changelog: Organization runner controls for Copilot cloud agent

4. Per-user CLI activity now visible in org usage reports

Org admins can now see which individual users are active on Copilot CLI, including session counts, token usage, and CLI version per user in 1-day and 28-day reports. Useful for adoption planning, cost allocation, and version upgrade rollouts.

→ Changelog: Copilot usage metrics now includes per-user GitHub Copilot CLI activity in organization reports

5. Copilot cloud agent now signs its commits

Every commit made by Copilot cloud agent is now cryptographically signed, appearing as Verified on GitHub. Admins no longer need to exempt the agent from "Require signed commits" branch protection rules or rulesets.

→ Changelog: Copilot cloud agent signs its commits
